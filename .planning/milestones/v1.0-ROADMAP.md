# Milestone v1.0: Infracode WhatsApp Platform — Production

**Status:** ✅ SHIPPED 2026-04-25
**Phases:** 1–8
**Total Plans:** 34

## Overview

Primeira versão de produção da plataforma SaaS multi-tenant para automação de atendimento via WhatsApp. O milestone transformou um protótipo funcional em um sistema pronto para clientes reais: segurança corrigida, CRM com dados reais, ciclo de vida de sessão formalizado, classificador de intenção LLM, métricas de atendimento, admin via WhatsApp, e módulo de aprendizado contínuo com degradação graciosa.

## Phases

### Phase 1: Security Hardening

**Goal**: O codebase é deployável a qualquer ambiente sem exposição de CSRF, bypass de autenticação, vazamento de credenciais ou API keys em plaintext — barra mínima antes de qualquer conta de cliente real.

**Depends on**: Nada (primeira fase — deve ser completada antes do onboarding de produção)

**Plans**: 4 plans

- [x] 01-01-PLAN.md — Wave 0: Test scaffolds para todos os 9 comportamentos de segurança (SEC-01..SEC-04)
- [x] 01-02-PLAN.md — Wave 1: CORS allowlist + auth bypass guard (SEC-01, SEC-02)
- [x] 01-03-PLAN.md — Wave 1: aiFallbackApiKey encryption + migration (SEC-03)
- [x] 01-04-PLAN.md — Wave 1: Session files assertion + query-string token removal (SEC-04)

**Details:**
4 fixes de segurança críticos pré-produção: CORS com allowlist explícita, auth bypass restrito a `NODE_ENV=development`, criptografia do `aiFallbackApiKey` com o mesmo padrão do `aiApiKeyEncrypted`, e arquivos de sessão WhatsApp fora do repositório com startup assertion.

## Milestone Summary

**Key Accomplishments:**
1. 4 correções de segurança críticas pré-produção (CORS, auth bypass, criptografia, session files)
2. CRM com LID/JID normalizado, campos customizados, tags e histórico cross-session
3. `AdminIdentityService` — único ponto de identificação de admin; classe de bug "admin como cliente" eliminada
4. Ciclo de vida de sessão formalizado com estados Redis+PostgreSQL, BullMQ timeouts, `InstanceEventBus`
5. Chatbot com classificador de intenção Groq LLM e transições automáticas (encerramento, urgência, handoff)
6. Admin gerencia atendimento pelo WhatsApp: documentos, status, métricas, log de auditoria completo

**Key Decisions:**
- Null Object pattern para `aprendizadoContinuo` — módulo degradável sem `isEnabled` guards
- `InstanceEventBus` como seam de desacoplamento (não big-bang rewrite do InstanceOrchestrator)
- BullMQ deduplication `extend:true` para session timeouts — O(1) reset em atividade
- Redis (live) + PostgreSQL (durable) para estado de sessão — nunca apenas Map in-process
- Base64 com gate 5 MB para document dispatch — Baileys não suporta file:// URLs locais

**Issues Resolved:**
- Bug LID/JID: número real exibido em todo o CRM
- Admin sendo tratado como cliente em alguns fluxos — eliminado arquiteturalmente
- `aiFallbackApiKey` em plaintext no banco — criptografado
- CORS com `origin: true` — corrigido com allowlist
- Bug humanTakeover: bot respondia mensagens do próprio atendente (corrigido no post-audit bugfix)

**Issues Deferred (Tech Debt):**
- VERIFICATION.md retroativos para fases 2, 3, 4, 6, 8 (fases têm SUMMARY.md com evidência)
- `SESSION_LIFECYCLE_V2` e `INTENT_CLASSIFIER_V2` ainda controlados por env var (backlog: Phase 999.1)
- Phase 8 Plan 04 Task 3: urgency score dashboard wiring — human verification pending
- 0/47 requirements formalmente marcados em REQUIREMENTS.md (traceability não foi atualizado durante execução)

**Technical Debt Incurred:**
- `InstanceOrchestrator` em service.ts ainda com ~5.200 linhas — extração incremental continua no v1.1
- Feature flags `SESSION_LIFECYCLE_V2` e `INTENT_CLASSIFIER_V2` como env vars (não painel)
