## formatPhone() Output Contract

This is the core deliverable of Plan 2.2. The executor MUST implement exactly these output formats.

### Input Classes and Required Output

| Input Class | Example Input | Required Output | Rule |
|-------------|---------------|-----------------|------|
| BR mobile, 11-digit local | `556892549342` | `+55 68 92549-342` | Strip country code → 11 digits → `+55 {DDD} {5d}-{4d}` |
| BR mobile, 11-digit with `+` | `+5511987654321` | `+55 11 98765-4321` | Same |
| BR landline, 10-digit local | `5511987654320` → 10 local digits | `+55 11 9876-5432` | Strip country code → 10 digits → `+55 {DDD} {4d}-{4d}` |
| E.164 international (non-BR) | `+14155550199` | `+14155550199` | Return as-is — no formatting applied |
| Null / unresolvable | `null` | `"Aguardando número"` | phoneNumber is null (LID pending) |
| Raw JID suffix present | `5511@c.us` | Strip suffix first, then format | `toPhone()` strips `@*` before formatting |
| Permanently unresolvable | non-numeric garbage after stripping | `"Contato desconhecido"` | Cannot produce valid E.164 |

### Detailed Format Strings

```
BR 11-digit local:  +55 {DDD} {5 digits}-{4 digits}
                    Example: +55 68 92549-342   ← note: 5+3 split for 9-digit number
                    Corrected: +55 {DDD} {5d}{4d} → +55 11 98765-4321

BR 10-digit local:  +55 {DDD} {4 digits}-{4 digits}
                    Example: +55 11 9876-5432

International E.164: returned verbatim with leading +
                    Example: +14155550199
```

**Clarification on BR 11-digit:** local = `digits after stripping country code 55`. If local.length === 11: `+55 ${local.slice(0,2)} ${local.slice(2,7)}-${local.slice(7)}` → produces `+55 11 98765-4321`.
If local.length === 10: `+55 ${local.slice(0,2)} ${local.slice(2,6)}-${local.slice(6)}` → produces `+55 11 9876-5432`.

**The existing `formatPhone()` in crm-screen.tsx uses `(DDD)` format** — the new utility switches to `+55 DDD` prefix format (with country code prefix explicit) for consistency with E.164 convention. The CONTEXT.md decision specifies: `+55 11 99999-9999`. The executor must update the existing helper to match this format.

## Surfaces That Must Apply formatPhone()

Exhaustive list — every location where a contact identifier appears in the operator UI:

| Surface | Location in Code | Field Rendered | Required Treatment |
|---------|-----------------|----------------|-------------------|
| CRM contact list | `ContactCard` sub-line (`<p className="text-xs">`) | `c.phoneNumber` | `formatPhone(c.phoneNumber)` or "Aguardando número" if null |
| CRM contact list | `ContactCard` name fallback | `c.displayName` when null and not LID | `formatPhone(c.phoneNumber)` |
| Chat thread header | Contact detail panel header | `phoneNumber` | `formatPhone()` |
| Contact detail panel | "Telefone" field in side panel | `phoneNumber` | `formatPhone()` |
| Message send path | `normalizePhoneForSend()` for API call | `c.jid` or `c.phoneNumber` | Must use resolved `phoneNumber`, never raw JID/LID |
| Search results | Contact matches in search | `phoneNumber` | `formatPhone()` |
| Metrics/reports (if phone shown) | Any table or summary showing phone | `phoneNumber` | `formatPhone()` |

**Absolute rule:** The string `@lid`, `@c.us`, or any raw JID must never appear in any rendered text node in the operator-facing panel. The executor must audit all `.tsx` string interpolations that touch `jid` or `phoneNumber`.

## Placeholder Styling Spec

When `phoneNumber` is null, the sub-line in ContactCard renders "Aguardando número". The executor applies this exact class combination:

```tsx
// Sub-line when phoneNumber is null
<p className="text-xs text-[var(--text-tertiary)] truncate italic">
  Aguardando número
</p>

// Sub-line when phoneNumber is present and formatted
<p className="text-xs text-[var(--text-tertiary)] truncate">
  {formatPhone(phoneNumber)}
</p>

// Sub-line when permanently unresolvable
<p className="text-xs text-[var(--text-tertiary)] truncate">
  Contato desconhecido
</p>
```

**Italic** differentiates the transient placeholder from real data without introducing a new color. This is the only styling difference between placeholder and real phone data.

## Accessibility Requirements

| Concern | Requirement |
|---------|-------------|
| Placeholder text contrast | `--text-tertiary` (#5a5a5a on #111111) = 3.2:1 contrast ratio. Acceptable for supplementary sub-line text (WCAG AA large text threshold). The italic style aids visual distinction without relying on color alone. |
| Phone number screen reader | `formatPhone()` output is plain text — screen readers will read it correctly as digits and punctuation. No `aria-label` override needed. |
| "Aguardando número" screen reader | The placeholder string is meaningful Portuguese — screen readers will announce it correctly. No additional `aria-label` needed. |
| Raw JID must not reach DOM | Any `@lid` or `@c.us` string reaching a rendered text node is an accessibility and data leak failure. |
