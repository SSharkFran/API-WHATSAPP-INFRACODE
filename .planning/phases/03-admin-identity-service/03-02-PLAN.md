<objective>
At Baileys connection open, resolve the admin phone to a current JID via sock.onWhatsApp()
and cache it in Redis as `instance:{instanceId}:admin_jid`. When a `@lid` message arrives,
the AdminIdentityService can match it against the cached JID. Clear the cache on disconnect.

Purpose: Eliminates @lid false-negative admin identification (ADM-03) without requiring
synchronous LID resolution in the message handling path.

Output:
- baileys-session.worker.ts modified — adminPhone in WorkerInitPayload; onWhatsApp() call at connection open; admin-jid-resolved message sent to parent
- service.ts modified — admin-jid-resolved handler writes to Redis; startWorkerForInstance() passes adminPhone; disconnect handler deletes Redis key; AdminIdentityInput includes cachedAdminJid
- admin-identity.service.test.ts updated — Scenario 4 now verifies @lid matches via cachedAdminJid input
</objective>

<threat_model>
## Threats

| Threat | Severity | Mitigation |
|--------|----------|-----------|
| Stale admin JID in Redis after WhatsApp re-registration (Spoofing — ADM-03) | High | Redis key is DEL'd on DISCONNECTED/PAUSED status events; repopulated on next connection open |
| TTL accidentally set on admin_jid Redis key, causing mid-session expiry | High | Task 1 uses `redis.set(key, value)` with no TTL argument — NOT `redis.set(key, value, "EX", N)` |
| admin-jid-resolved message forged by malicious code in worker thread | Medium | Worker thread is a trusted internal process spawned by the orchestrator — no external input reaches this message type |
| onWhatsApp() call at connection open fails or throws, crashing the worker | Medium | Try/catch wraps the onWhatsApp() call with log("warn") — failure is non-fatal, cache just stays empty |
| Multiple adminCandidatePhones — only primary is cached | Low | Task 1 caches the first resolved JID (for leadsPhoneNumber); secondary (adminAlertPhone) would need a separate key — accepted risk for v1, fallback to phone match covers most cases |
</threat_model>
